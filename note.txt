================================================================
SECTION 1: HOW THE PROJECT WAS PREPARED FOR DEPLOYMENT
================================================================

PROBLEM BEFORE PREPARATION
---------------------------
The project was only set up to run locally. Several things would
break when deployed to real servers:

  1. No .gitignore → node_modules, .env secrets, and local uploads
     would all be pushed to GitHub accidentally.

  2. React Router would break on Vercel → navigating directly to
     a URL like /dashboard or /register/123 would return a 404
     because Vercel doesn't know to serve index.html for those paths.

  3. CORS was set to origin: "*" → this means ANY website on the
     internet could make requests to the backend API, which is a
     security risk.

  4. Image uploads used local disk storage (backend/uploads/) →
     Render (the hosting platform) has an ephemeral filesystem,
     meaning every time the server restarts or redeploys, all
     uploaded files are permanently deleted.

  5. Environment variables were hardcoded in some places → the app
     would not know where to find the backend once deployed.


WHAT WAS DONE — STEP BY STEP
-----------------------------

STEP 1: Created .gitignore (root of project)
  File: .gitignore
  Purpose: Prevent sensitive and unnecessary files from being
           pushed to GitHub.
  What it excludes:
    - node_modules/         (large, auto-generated, not needed in git)
    - .env                  (contains secret keys — NEVER push this)
    - dist/ and build/      (frontend build output, regenerated on deploy)
    - backend/uploads/*     (user-uploaded images, handled by Cloudinary now)
    - *.log files
    - OS files (.DS_Store, Thumbs.db)
  Special: backend/uploads/.gitkeep is still tracked so the folder
           exists on the server when it first starts up.


STEP 2: Created frontend/vercel.json
  File: frontend/vercel.json
  Purpose: Fix React Router on Vercel.
  Content:
    {
      "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }]
    }
  Why needed: Vercel serves static files. When a user visits
  /dashboard directly, Vercel looks for a file called dashboard —
  which doesn't exist. This rule tells Vercel to always serve
  index.html and let React Router handle the routing client-side.
  Without this file, every page except "/" would return a 404.


STEP 3: Fixed CORS in backend/server.js
  File: backend/server.js
  Before:
    cors({ origin: "*" })   ← allows ALL websites to call your API

  After:
    const allowedOrigins = process.env.CLIENT_URL
      ? process.env.CLIENT_URL.split(",").map((o) => o.trim())
      : ["http://localhost:5173"];

    cors({
      origin: (origin, callback) => {
        if (!origin) return callback(null, true);
        if (allowedOrigins.includes(origin)) return callback(null, true);
        callback(new Error(`CORS: origin ${origin} not allowed`));
      }
    })

  How it works:
    - Reads CLIENT_URL environment variable from .env
    - If not set, defaults to localhost:5173 (local dev)
    - Supports comma-separated list for multiple origins
    - Requests with no origin (mobile apps, Render health checks,
      curl) are still allowed through
  What to do on Render:
    - Set CLIENT_URL = https://your-app.vercel.app


STEP 4: Created .env.example files
  Files: backend/.env.example and frontend/.env.example
  Purpose: Document what environment variables are needed without
           exposing the actual secret values.
  These files ARE committed to git. The real .env files are NOT.

  backend/.env.example contains:
    MONGO_URI=mongodb+srv://...
    PORT=5000
    JWT_SECRET=replace_with_a_long_random_secret
    CLIENT_URL=http://localhost:5173
    CLOUDINARY_CLOUD_NAME=your_cloud_name
    CLOUDINARY_API_KEY=your_api_key
    CLOUDINARY_API_SECRET=your_api_secret

  frontend/.env.example contains:
    VITE_API_URL=http://localhost:5000


STEP 5: Added VITE_API_URL to frontend
  File: frontend/.env
  Content: VITE_API_URL=http://localhost:5000
  Purpose: All frontend API calls use this variable so that
           switching from localhost to the live Render URL only
           requires changing one value in Vercel's dashboard.
  Used in: frontend/src/api.js as the axios baseURL.

SECTION 2: HOW CLOUDINARY WAS IMPLEMENTED
================================================================

PROBLEM
-------
The original code used multer with disk storage:
  - Uploaded images were saved to backend/uploads/ folder
  - This folder lives on the server's local hard drive
  - Render (and most cloud platforms) have ephemeral filesystems —
    files written to disk disappear on every restart or redeploy
  - This meant event banners would vanish every time the server
    restarted, breaking the entire app

SOLUTION
--------
Replace local disk storage with Cloudinary — a cloud media hosting
service. Images are uploaded directly to Cloudinary's servers and
stored permanently there. The database stores the full Cloudinary
URL instead of a local file path.


PACKAGES INSTALLED
------------------
  npm install cloudinary multer-storage-cloudinary

  - cloudinary        : Official Cloudinary Node.js SDK
  - multer-storage-cloudinary : Connects multer (file upload handler)
                                to Cloudinary as the storage destination


FILES CREATED / CHANGED
------------------------

1. NEW FILE: backend/config/cloudinary.js
   Purpose: Initializes and configures the Cloudinary SDK using
            environment variables.
   Code:
     const cloudinary = require("cloudinary").v2;
     cloudinary.config({
       cloud_name: process.env.CLOUDINARY_CLOUD_NAME,
       api_key:    process.env.CLOUDINARY_API_KEY,
       api_secret: process.env.CLOUDINARY_API_SECRET,
     });
     module.exports = cloudinary;


2. CHANGED: backend/middleware/uploadMiddleware.js
   Before (disk storage):
     const storage = multer.diskStorage({
       destination: (req, file, cb) => { cb(null, "uploads/"); },
       filename:    (req, file, cb) => { cb(null, uniqueName); },
     });

   After (Cloudinary storage):
     const { CloudinaryStorage } = require("multer-storage-cloudinary");
     const cloudinary = require("../config/cloudinary");

     const storage = new CloudinaryStorage({
       cloudinary,
       params: {
         folder: "lhpyaya-events",       ← images go in this folder on Cloudinary
         allowed_formats: ["jpg", "jpeg", "png", "webp"],
         transformation: [{ quality: "auto", fetch_format: "auto" }],
                                          ← auto-optimize quality and format
       },
     });

   Everything else (fileFilter, size limit, export) stayed the same.


3. CHANGED: backend/controllers/eventController.js
   Before:
     image: req.file ? req.file.path.replace(/\\/g, "/") : ""

   After:
     image: req.file ? req.file.path : ""

   Why: With disk storage, req.file.path was a local file path like
        "uploads\image.jpg" with Windows backslashes that needed fixing.
        With Cloudinary storage, req.file.path is already a full HTTPS
        URL like "https://res.cloudinary.com/dyrteayr3/image/upload/..."
        so no fixing is needed. Same change applied to updateEvent too.


4. CHANGED: backend/server.js
   Removed:
     const path = require("path");
     app.use("/uploads", express.static(path.join(__dirname, "uploads")));

   Why: The /uploads static route was serving local files from disk.
        Now that images are on Cloudinary, there are no local files to
        serve. This line was removed to keep the code clean.


5. CHANGED: All frontend pages that display event images
   Files:
     - frontend/src/pages/LandingPage.jsx
     - frontend/src/pages/AdminDashboard.jsx
     - frontend/src/pages/RegisterEvent.jsx
     - frontend/src/pages/EditEvent.jsx

   Before (constructing URL from backend base + relative path):
     src={`${import.meta.env.VITE_API_URL}/${event.image}`}
     ← e.g. "http://localhost:5000/uploads/image.jpg"

   After (using URL directly from database):
     src={event.image}
     ← e.g. "https://res.cloudinary.com/dyrteayr3/image/upload/..."

   Why: The database now stores the full Cloudinary URL, so there is
        no need to prepend anything. The image URL works from anywhere
        — localhost, Render, or any device in the world.


6. CHANGED: backend/.env (and .env.example)
   Added three new variables:
     CLOUDINARY_CLOUD_NAME=dyrteayr3
     CLOUDINARY_API_KEY=293196834383469
     CLOUDINARY_API_SECRET=NCUWjCYmtdiTi3Wqdbr5afYvNWI

   These are your actual credentials from cloudinary.com/console.
   You MUST also add these same three variables to your Render
   environment settings — otherwise image uploads will fail in
   production.

SECTION 3: HOW IMAGE UPLOAD FLOW WORKS NOW
================================================================

  1. Admin selects an image file in CreateEvent or EditEvent form
  2. Frontend sends FormData (including the image) to POST /api/events
  3. Express receives the request and runs the multer upload middleware
  4. multer-storage-cloudinary intercepts the file and uploads it
     directly to Cloudinary — the image NEVER touches the server disk
  5. Cloudinary returns a secure HTTPS URL
  6. req.file.path = "https://res.cloudinary.com/..."
  7. This URL is saved into MongoDB as the event's image field
  8. When any page loads events, it receives the full URL from MongoDB
  9. <img src={event.image} /> loads directly from Cloudinary's CDN
