===================================================
FRONTEND — LOGIN & SIGNUP PAGES
===================================================

FILES CHANGED
-------------
  src/pages/LoginPage.jsx   — filled with form UI + login logic
  src/pages/SignupPage.jsx  — filled with form UI + signup logic

Both pages follow the same pattern: controlled form state → submit → API call →
call login() from AuthContext → redirect.

---------------------------------------------------
1. LOGINPAGE.JSX
---------------------------------------------------
STATE
  form    — object holding { email, password }, updated on every keystroke via handleChange.
  loading — boolean, true while the API call is in flight (disables the button).

FLOW
  1. User fills in email and password.
  2. On submit, e.preventDefault() stops the page from refreshing.
  3. Calls POST /api/auth/login with the form data via the api instance.
  4. On success:
       - Calls login(token, user) from AuthContext, which saves the token to
         localStorage and sets the user state globally.
       - Shows a success toast: "Welcome back, <name>!"
       - Redirects the user. If they were redirected to /login from a protected
         route (e.g. they tried to go to /dashboard while logged out), they are
         sent back to that original page using location.state?.from?.pathname.
         Otherwise they go to /dashboard.
  5. On error: shows the server's error message via toast (e.g. "Invalid email or password").
  6. loading is always reset to false in the finally block.

UI
  - Clean centered card with email + password inputs.
  - Submit button shows "Signing in..." while loading and is disabled to prevent double-submit.
  - Link to /signup at the bottom.
  - Focus ring uses the brand color (#ff9324).

---------------------------------------------------
2. SIGNUPPAGE.JSX
---------------------------------------------------
STATE
  form    — object holding { name, email, password, confirmPassword }.
  loading — boolean for the API call.

FLOW
  1. Client-side password match check before sending the request:
       if (form.password !== form.confirmPassword) → toast error, return early.
     This avoids an unnecessary API call for a simple mismatch.
  2. Calls POST /api/auth/register with { name, email, password }.
     (confirmPassword is NOT sent to the backend — it's only for client validation.)
  3. On success:
       - Calls login(token, user) — same as login page, immediately logs the user in.
       - Shows a success toast: "Account created! Welcome, <name>!"
       - Redirects to /dashboard.
  4. On error: shows server error toast (e.g. "User already exists with this email").

UI
  - Same card layout as LoginPage for visual consistency.
  - Four fields: Full Name, Email, Password, Confirm Password.
  - Submit button shows "Creating account..." while loading.
  - Link to /login at the bottom.

---------------------------------------------------
SHARED PATTERNS
---------------------------------------------------
handleChange
  Uses the spread pattern to update only the changed field:
    setForm({ ...form, [e.target.name]: e.target.value })
  The input's name attribute must match the key in the form state object.

Error handling
  err.response?.data?.message uses optional chaining to safely read the
  server's error message. Falls back to a generic message if unavailable.

toast (react-hot-toast)
  toast.success() and toast.error() show non-blocking notification popups.
  The Toaster component in main.jsx renders them in the bottom-right corner.
