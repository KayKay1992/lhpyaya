===================================================
CONTROLLERS — EXPLANATION
===================================================

Controllers contain the actual business logic for each API endpoint. Each function
receives the request (req) and response (res) objects from Express, does its work
(validation, database query, etc.), and sends back a JSON response.

---------------------------------------------------
1. AUTH CONTROLLER (controllers/authController.js)
---------------------------------------------------
Three functions handle user authentication:

  registerUser — POST /api/auth/register
    - Validates that name, email, and password are provided.
    - Checks if a user with that email already exists (returns 400 if so).
    - Hashes the password using bcrypt (genSalt + hash) before saving — plain text
      passwords are never stored.
    - Creates the user in the database with the hashed password and role.
    - Signs a JWT token containing the user's id and role, with a 7-day expiry.
    - Returns the token and basic user info (no password).

  loginUser — POST /api/auth/login
    - Validates email and password are provided.
    - Finds the user by email. Returns 401 if not found (same message as wrong
      password — avoids leaking whether an email exists).
    - Uses bcrypt.compare() to check the submitted password against the hash.
    - Signs and returns a JWT token on success.

  getMe — GET /api/auth/me
    - Protected route: relies on authMiddleware to attach req.user.
    - Fetches the logged-in user's profile from the database using req.user.id.
    - Uses .select('-password') to exclude the password field from the response.

JWT_SECRET is read from process.env so it is never hardcoded in source code.

---------------------------------------------------
2. EVENT CONTROLLER (controllers/eventController.js)
---------------------------------------------------
Five functions cover the full CRUD lifecycle for events:

  getEvents — GET /api/events  (Public)
    - Fetches all events from the database, sorted by date ascending (soonest first).

  getEventById — GET /api/events/:id  (Public)
    - Fetches a single event by its MongoDB _id from the URL param.
    - Returns 404 if no event is found with that id.

  createEvent — POST /api/events  (Admin only)
    - Validates all required fields (title, description, date, time, location).
    - If a file was uploaded (via multer), stores req.file.path as the image field.
    - Sets createdBy to req.user.id (the logged-in admin's id, attached by authMiddleware).

  updateEvent — PUT /api/events/:id  (Admin only)
    - Finds the event by id, returns 404 if missing.
    - Uses the "update only what was sent" pattern: falls back to the existing value
      if a field is not included in the request body (e.g. event.title = title || event.title).
    - Saves and returns the updated document.

  deleteEvent — DELETE /api/events/:id  (Admin only)
    - Finds the event by id, returns 404 if missing.
    - Calls .deleteOne() on the document to remove it from the database.

---------------------------------------------------
3. REGISTRATION CONTROLLER (controllers/registrationController.js)
---------------------------------------------------
Two functions handle event registrations:

  registerForEvent — POST /api/registrations/:eventId  (Public)
    - Validates name and email are provided.
    - Checks the event exists (returns 404 if not).
    - Checks for a duplicate registration: if the same email is already registered
      for that event, returns 400 — prevents double sign-ups.
    - Creates and saves the registration document linked to the event via ObjectId.

  getRegistrantsByEvent — GET /api/registrations/:eventId  (Admin only)
    - Checks the event exists.
    - Fetches all registrations for that event, sorted newest first.
    - Returns the event title, total registrant count, and the full list — giving
      the admin a clean summary view.

---------------------------------------------------
ERROR HANDLING PATTERN
---------------------------------------------------
Every controller function wraps database calls in try/catch.
- Validation errors return 400 (bad request).
- Not-found cases return 404.
- Auth failures return 401.
- Unexpected server errors return 500 with the error message for debugging.
