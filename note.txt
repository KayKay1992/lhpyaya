BUG FIX: Add Speaker Returns 500 Error
----------------------------------------------------
Problem:
  Clicking "Add Speaker" on the Edit Event page caused a
  500 Internal Server Error from the backend.

Root Cause:
  The API call was manually setting the HTTP header:
    Content-Type: multipart/form-data
  BUT without including the multipart boundary string.
  A correct multipart Content-Type looks like:
    Content-Type: multipart/form-data; boundary=----XYZ123
  Without the boundary, the server's multer middleware
  cannot identify where each form field starts and ends
  in the request body. This caused multer to throw an
  error before the controller even ran, resulting in a 500.

Fix:
  Removed the manual Content-Type header override from
  the speaker API call in EditEvent.jsx. Axios and the
  browser automatically set the correct Content-Type with
  the proper boundary when FormData is used — so no manual
  header is needed.

  Before:
    api.post(`/events/${id}/speakers`, fd, {
      headers: { "Content-Type": "multipart/form-data" },
    })

  After:
    api.post(`/events/${id}/speakers`, fd)

File: frontend/src/pages/EditEvent.jsx

BUG FIX: Event Date Shows 1 Day Earlier Than Selected
----------------------------------------------------
Problem:
  Admin selected Saturday 27 June as the event date,
  but the registration page displayed Friday 26 June.

Root Cause:
  MongoDB stores dates as UTC. When the admin picks
  "2026-06-27" in the date input, it gets stored as:
    2026-06-27T00:00:00.000Z  (UTC midnight)
  When JavaScript's new Date() parses an ISO UTC string
  and toLocaleDateString() renders it, the conversion to
  local time can shift the date back by one day in certain
  server or browser timezone environments (e.g. UTC+0 edge
  cases, or environments where UTC midnight becomes the
  previous evening in local time).

Fix:
  Replaced:
    new Date(eventDetails.date)
  With:
    new Date(eventDetails.date.slice(0, 10) + "T12:00:00")

  By slicing just the date portion (YYYY-MM-DD) and
  appending T12:00:00 (noon), the Date object is created
  at midday local time. A timezone offset of up to ±12
  hours in either direction can never shift noon into a
  different calendar day — so the correct date always
  displays regardless of the user's timezone.

  This fix was applied to both places the date is shown:
    1. The description card pill badge (registration page)
    2. The event details card on the success screen

File: frontend/src/pages/RegisterEvent.jsx
